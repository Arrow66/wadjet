
# 🧠 RemoteVerify: Multi-Agent Architecture Guide

RemoteVerify uses a highly parallel **LangGraph multi-agent ensemble** to analyze remote job listings. Rather than relying on a single Large Language Model (which is prone to hallucinations), the system isolates specific investigative tasks into specialized "Agents."

These agents act as an autonomous fraud-detection task force. This document details how each agent behaves, examples of their system prompts, and the mathematics behind the final TrustScore calculation.

---

## 🛡️ Agent Roster & Behaviors

### Agent 1: Linguistic Forensics
**Goal:** Detect manipulative phrasing, artificial urgency, and "low-barrier" hooks.
**Behavior:** This agent reads the raw job description and looks for classic scam tells: demanding payment upfront, explicitly stating "no experience required" for senior roles, or using excessive capitalization to induce urgency.
**Example Prompt Snippet:**
> "Analyze the text for manipulation or scam patterns: Urgency (Act now, immediate hire), Low-Barrier (No experience needed, everyone approved), or Grammar/Professionalism. Base your analysis STRICTLY on the text provided."

### Agent 2: Company Intelligence (Highest Weight)
**Goal:** Verify the corporate entity actually exists.
**Behavior:** This agent is strictly grounded using the **Google Search tool**. It is forbidden from guessing; it must search the live internet for official websites, LinkedIn pages, and news mentions. It also cross-references the job title against the company's known industry to detect impersonation scams.
**Example Prompt Snippet:**
> "You MUST use Google Search to verify this. Look for: Their official website, a LinkedIn company page, and News mentions. Also check if the job role makes sense for this company's industry to detect impersonation scams."

### Agent 3: Compensation Analyst
**Goal:** Identify "Too Good To Be True" financial claims.
**Behavior:** Compares the listed salary against market medians for the specified role and requirements.
**Example Prompt Snippet:**
> "Determine if the compensation is 'Too Good to be True'. If the salary is >2x the median for the listed job title and requirements (e.g., $100k for entry-level data entry), flag it with a 90+ score."

### Agent 4: Digital Footprint
**Goal:** Analyze domain history and contact infrastructure.
**Behavior:** Also equipped with **Google Search Grounding**, this agent investigates the domains associated with the company and contact emails. It looks for extremely recent domain registrations (a massive red flag) and WHOIS privacy shielding.
**Example Prompt Snippet:**
> "You MUST use Google Search to find domain registration information (WHOIS data) for the company. Look for: The year the domain was registered. Extremely new domains (< 6 months) combined with high-paying roles are a major red flag."

### Agent 5: Posting Pattern Analyst
**Goal:** Match the listing's structural footprint against known scam databases.
**Behavior:** Operates like an antivirus scanner. It compares the structure and flow of the listing to known fraud templates like Advance-fee scams, Reshipping/Package Mule scams, and Pyramid data-entry schemes.
**Example Prompt Snippet:**
> "Compare this job listing against known scam templates: Advance-fee fraud (requires buying equipment), Package Mule (receiving and forwarding packages), or Phishing (requires downloading strange apps)."

### Agent 6: The Adversary (Devil's Advocate)
**Goal:** Challenge the consensus to prevent group-think and hallucinations.
**Behavior:** After Agents 1-5 finish, the Adversary looks at their average score. It then takes the *opposite* position. If the agents think it's a scam, the Adversary uses Google Search to frantically find proof that it's actually a real startup. If it finds strong counter-evidence, it overrides the score.
**Example Prompt Snippet:**
> "YOUR MISSION: PLAY DEVIL'S ADVOCATE. You must take the opposite position. If you are arguing LEGITIMATE: search for proof the company is real. If you are arguing SUSPICIOUS: search for 'company name scam'."

## 🧮 How the TrustScore is Calculated

The final `TrustScore` is generated by the **Scorer Node**, which acts as the judge for the task force. 

### 1. Raw Weighted Average
First, Agents 1 through 5 each return a `riskScore` from 0 to 100 (where 100 = Guaranteed Scam). The Scorer Node applies a weighted average, prioritizing hard corporate evidence over soft linguistic clues:

- **Company Intel:** `30%`
- **Linguistic Forensics:** `20%`
- **Pattern Match:** `20%`
- **Compensation:** `15%`
- **Digital Footprint:** `15%`

*Example: If the Linguistic agent gives it an 80 (bad grammar) but the Company Intel agent gives it a 0 (huge public footprint), the Company score weighs heavier.*

### 2. The Adversarial Adjustment
Next, the system looks at the output of **Agent 6 (The Adversary)**. 
If the Adversary successfully found undeniable counter-evidence that proves the other 5 agents wrong, it applies a `confidenceAdjustment`. This adjustment can swing the final Risk Score by **-20 to +20 points**.

### 3. Final Conversion
Finally, the system converts the Risk Score into a user-friendly **TrustScore** by flipping the scale:
`TrustScore = 100 - Final Risk Score`

### 4. Tier & Confidence Mapping
The final TrustScore is mapped to UI tiers:
- **75 to 100:** `Verified` (Green)
- **40 to 74:** `Caution` (Yellow)
- **0 to 39:** `Warning` (Red)

> [!TIP]
> **Conflicting Signals:** If the agents wildly disagree (e.g., the spread between the highest and lowest agent score is greater than 60 points), the Scorer Node automatically downgrades the final Confidence Level to *"Medium Confidence (Conflicting Signals)"* to warn the user.
